About PRBench

A physical reasoning benchmark for robotics.

This is a work in progress! We are sharing it with you to get feedback so that we can determine how to focus our next efforts. Thanks for your help!

PRBench is designed to test "mid-level" robot reasoning. Existing benchmarks focus on either long-horizon high-level semantic understanding ("create a meal from existing ingredients"), or short-horizon low-level perception and control (pixels to torques). PRBench instead considers the challenges that arise when these two levels meet, and when they cannot be easily disentangled: the messy chain of intermediate constraints required to actually realize a high-level goal ("create a "meal"), e.g., retrieving tools from cluttered drawers, opening stuck cabinets, reaching around obstacles, manipulating ingredients in tight spaces, and sequencing these motions so nothing falls, spills, or blocks future actions. In general, these are tasks that feature:

  • Tight constraints (spatial, kinematic, dynamic, collision, force, among others)
  • Varying numbers of objects
  • Long time horizons
  • Sparse rewards
  • Diverse task distributions, requiring generalization
  • Challenges inspired by the task-and-motion-planning (TAMP) literature

PRBench is implemented as a collection of Gymnasium environments. Performance is evaluated based on not only task performance (rewards), but also training time and sample complexity (for data-driven approaches) and evaluation time (especially for planning-based approaches). Our goal is to facilitate direct comparisons between many different approaches, evaluating and supporting progress in the state of the art for robot physical reasoning.

What Makes PRBench Challenging? ▼

For Reinforcement Learning

Environments have long horizons and sparse rewards. Users are welcome to engineer dense rewards, but doing so may be nontrivial. Environments also have very diverse task distributions, so learned policies must generalize.

For Imitation Learning

Physical reasoning requires understanding physical constraints (e.g., spatial relationships, kinematics, dynamics). Imitating surface-level patterns in demonstrations is not enough to generalize to broad task distributions.

For Vision-Language Models

The physical reasoning required in PRBench is not easy to represent in natural language. Spatial reasonining, a subset of physical reasoning, is a known challenge for vision-language models.

For Hierarchical Approaches

Approaches that first decide "what to do" and then decide "how to do it" -- whether in hierarchical RL, planning, or with multi-level foundation models -- will run into difficulties when there are couplings between these high-level and low-level decisions.

For Task and Motion Planning

PRBench does not provide any models for TAMP. Users are welcome to engineer their own, but doing so may be nontrivial. Some environments contain many objects, which may make planning slow.

For Human Engineers

PRBench features diverse task distributions and long time horizons, making it challenging for engineers to hand-design even very environment-specific solutions. A single task may be straightforward, but designing generalized solutions is nontrivial.

Usage

PRBench is not ready for public use yet (it is under development here). These instructions show how it will work when it is ready.

Installation

PRBench will be available on PyPI when it is ready. At that time, you will be able to install it with pip install prbench. Until then, you can install it from source as follows. First, we strongly recommend uv. Then choose one of the following based on your needs:

# Core dependencies only
uv pip install -r optional_prpl_requirements/core.txt && uv pip install -e .

# Everything (excluding develop)
uv pip install -e ".[all]"

# Specific environments
uv pip install -e ".[geom2d]"      # Geometric 2D only
uv pip install -e ".[dynamic2d]"   # Dynamic 2D only
uv pip install -e ".[geom3d]"      # Geometric 3D only
uv pip install -e ".[tidybot]"     # TidyBot only

# Compositional
uv pip install -e ".[geom2d,geom3d]"

Basic Usage (Gym API)

import prbench
prbench.register_all_environments()
env = prbench.make("prbench/Obstruction2D-o3-v0")  # 3 obstructions
obs, info = env.reset()  # procedural generation
action = env.action_space.sample()
next_obs, reward, terminated, truncated, info = env.step(action)
img = env.render()

Object-Centric States ▼

All environments in PRBench use object-centric states:

from prbench.envs.geom2d.obstruction2d import ObjectCentricObstruction2DEnv
env = ObjectCentricObstruction2DEnv(num_obstructions=3)
obs, _ = env.reset(seed=123)
print(obs.pretty_str())

For compatibility with baselines, observations are provided as vectors. Convert between vectors and object-centric states:

import prbench
prbench.register_all_environments()
env = prbench.make("prbench/Obstruction2D-o3-v0")
vec_obs, _ = env.reset(seed=123)
object_centric_obs = env.observation_space.devectorize(vec_obs)
recovered_vec_obs = env.observation_space.vectorize(object_centric_obs)

Environments

PRBench environments are organized into four categories. Click on any environment family to explore variants and details.

Geometric 2D

2D environments focused on geometric reasoning and spatial relationships.

  • ClutteredRetrieval2D 3 variants
  • ClutteredStorage2D 4 variants
  • Motion2D 6 variants
  • Obstruction2D 5 variants
  • PushPullHook2D 1 variant
  • StickButton2D 5 variants

Geometric 3D

3D environments for testing spatial reasoning in three dimensions.

  • Motion3D 1 variant
  • Obstruction3D 5 variants
  • Packing3D 3 variants

Dynamic 2D

2D environments involving dynamic physical interactions and motion.

  • DynObstruction2D 5 variants
  • DynPushPullHook2D 3 variants
  • DynPushT2D 1 variant
  • DynScoopPour2D 8 variants

Dynamic 3D

3D environments with complex dynamics and physical interactions.

  • TidyBot3D-base_motion 1 variant
  • TidyBot3D-cupboard-o12-ConstrainedFitting 1 variant
  • TidyBot3D-table-o20-SortClutteredBlocks 1 variant
